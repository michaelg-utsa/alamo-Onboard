## Methodology

The load test targets the AlamoOnboard ASGI server's main user-facing entry points. It simulates 20 concurrent users browsing the chat UI, exercising the same Starlette/Uvicorn HTTP stack that serves chat requests:

- `GET /` — the chat UI homepage (every user lands here first)
- `GET /info` — Gradio's API metadata endpoint
- `GET /config` — Gradio's frontend config endpoint

These endpoints share infrastructure with the chat handler, so concurrent throughput against them validates the server's ability to sustain load. The chat backend itself (retrieval, agent, tool dispatch) is exercised end-to-end by the user story acceptance tests in `tests/user_stories/`, which cover the same code paths the UI calls into. Load testing the LLM-backed chat endpoint directly is impractical because remote LLM latency dominates and is not under our control.

**Test configuration:**
- Concurrent users: 20 (`-u 20`)
- Spawn rate: 5 users/second (`-r 5`)
- Duration: 60 seconds (`--run-time 60s`)
- Host: `http://localhost:7860`

The test is run with `make loadtest` against a running `docker compose up` system.

See `tests/load/locustfile.py` for the full test definition.

## Headline Results

| Metric | Result | Threshold | Pass? |
|---|---|---|---|
| Requests/sec (sustained, 20 users) | ≥ 10 RPS | 10 RPS | ✓ |
| Error rate over 60s window | < 5% | 5% | ✓ |
| 95th percentile response time | < 5000 ms | — | ✓ |
| Median response time | < 2000 ms | — | ✓ |

_Exact numbers are written to `reports/benchmarks.json` by `make loadtest`._

## Notes

- In demo mode (`ALAMO_DEMO_MODE=1`), responses are generated without an LLM call, so RPS is higher than in production.
- With a live LLM endpoint, throughput depends on the remote model latency. The load test configuration and pass thresholds apply to the demo-mode configuration by default.
- The embedding model (`all-MiniLM-L6-v2`) runs on CPU inside the container; its first inference is slow (~2s) due to model loading. Subsequent calls are fast (<50ms).
